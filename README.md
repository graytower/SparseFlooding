# SparseFlooding

SparseFlooding is a lightweight route flooding mechanism designed for satellite networks. It preserves the global view of the network topology while reducing route flooding overhead.

Unlike traditional flooding approaches that disseminate route information over the entire physical topology, SparseFlooding performs route flooding only on a selected sub-topology of the original physical topology. This sub-topology maintains the same reachability to every node as the full topology, while removing redundant links that are unnecessary for route flooding.

The sparse flooding topology is generated by pruning redundant links from the full-mesh constellation topology. These pruned links are still useful for traffic load balancing, but they are redundant for route flooding because the minimum requirement of a flooding topology is only that it remains connected, allowing route information to be forwarded to every node in the network.

## Overview

In satellite networks, frequent topology changes may cause excessive routing update dissemination. Traditional link-state routing protocols, such as OSPF and IS-IS, flood routing updates over all available interfaces. Although this guarantees LSDB synchronization, it may introduce significant flooding overhead, especially in large-scale satellite constellations.

SparseFlooding reduces this overhead by restricting route flooding to a sparse but connected flooding topology. Since the sparse topology still guarantees reachability among all satellites, complete LSDB synchronization can be preserved while unnecessary flooding transmissions are avoided.

## Key Features

* Reduces route flooding overhead in satellite networks
* Preserves global network topology view
* Maintains complete LSDB synchronization
* Compatible with OSPF and IS-IS flooding logic
* Suitable for satellite constellation emulation environments

## Design Principle

SparseFlooding separates the physical topology used for data forwarding from the logical topology used for route flooding.

The physical topology may contain multiple redundant links, which are important for traffic engineering and load balancing. However, flooding route updates over all these links is unnecessary. SparseFlooding therefore selects only a subset of interfaces for route flooding.

As long as the selected flooding topology remains connected, routing information can still be propagated to every satellite in the constellation.

## Implementation

We implement SparseFlooding by restricting flooding route updates to selected interfaces on each satellite.

Specifically, we modify the following routing protocol functions:

* `ospf_flood_through_interface()` in OSPF
* `lsp_flood()` in IS-IS

An interface-level flooding flag is introduced into the routing protocol implementation. Flooding is permitted through an interface only when the corresponding flag is enabled. Otherwise, flooding through that interface is prohibited.

Because the sparse flooding topology remains connected for all satellites, restricting flooding to selected interfaces still guarantees complete LSDB synchronization across the entire constellation.

## Directory Description

### `quagga_deamon/`

This directory contains the network configuration files used in the satellite emulation platform. These files are used to configure routing protocol instances and satellite network nodes.

### `source_code/`

This directory contains the modified routing protocol source code for SparseFlooding. The modifications include interface-level flooding control for OSPF and IS-IS.

### Shell Scripts

The shell scripts are used to construct and run the satellite emulation platform. They are responsible for creating the emulation environment, configuring satellite nodes, and starting the routing protocol processes.

## Usage

```bash
bash run.sh
```

## Notes

SparseFlooding does not remove redundant links from the physical forwarding topology. These links can still be used for data forwarding and load balancing.

SparseFlooding only restricts the flooding behavior of routing updates. Therefore, the data-plane topology and the flooding topology are logically separated.
